# Gemini 要約：入出力・関連度・HTML 契約（実装用仕様）

この文書は [gemini-summary-handoff.md](../gemini-summary-handoff.md) の「要約品質と生成パイプライン」部分を、**第三者がそのまま実装・別チャットに渡せる粒度**で固定する。UI の見た目は [output/patent-digest-handoff-page.html](../output/patent-digest-handoff-page.html) に従う。

---

## 1. 成果物パスと責務分担

| 成果物 | パス |
|--------|------|
| 生成されるダイジェスト HTML（日付付き） | `output/digest-YYYY-MM-DD.html` |
| 手動テンプレ・UI 規約の参照 | `output/patent-digest-handoff-page.html` |

**責務の分離（推奨）**

- **Gemini（または任意 LLM）**: 日本語の要約テキスト（および特許向けの追加メタ説明）を **構造化 JSON** で返す。HTML の `<details>` ラッパーや Tailwind クラスは **書かせない**（テンプレ＋後処理で付与し、表示の一貫性と XSS 回避のため）。
- **パイプライン（filter 相当）**: キーワード／スコアリングで **関連度 ★3/4/5** を決定し、`data-relevance-stars` とピル・枠色用のクラスを **確定的に付与**する。必要なら LLM に `relevance_stars` を **入力として渡し**、要約のトーンが星と矛盾しないようにする（星を LLM が上書きしない）。

---

## 2. 入力スキーマ（記事 1 件あたり）

パイプラインからモデルへ渡すペイロードは **JSON の配列要素 1 件＝1 記事**とする。

### 2.1 ニュース向けフィールド

| フィールド | 型 | 必須 | 説明 |
|------------|-----|------|------|
| `article_id` | string | はい | 安定した ID（URL のハッシュや GUID）。ログ・差分用。 |
| `title` | string | はい | 記事タイトル。 |
| `excerpt` | string | いいえ | RSS 説明文・抜粋。無い場合は空文字。 |
| `source_name` | string | はい | 例: `Google News：化学業界`。 |
| `url` | string | はい | 正規の記事 URL（トラッキング除去済み推奨）。 |
| `published_at` | string (ISO8601) | いいえ | 無い場合は省略または空。 |
| `matched_keywords` | string[] | いいえ | フィルタでヒットしたキーワード。要約内で触れてよいが羅列だけにしない。 |
| `relevance_stars` | 3 \| 4 \| 5 | はい | **パイプライン確定値**。モデルは変更しない。 |
| `language` | string | はい | 常に `ja` を想定。 |

### 2.2 欠損時の扱い

- `excerpt` が空: モデルは **`title` と `source_name` のみ**から要約し、推測で事実を足さない。短い場合は「抜粋が短いため要点のみ」と前置きしてよい（1 文まで）。
- `published_at` 欠損: 要約に日付を書かない。
- `url` は検証済みであることを前提とし、モデルに **別 URL の捏造**を禁止する（後述）。

### 2.3 特許向けの追加フィールド（ニュースと差分）

特許 1 件では、次を **追加**する（観点は `commit-report-tool` の `patent-digest-handoff.md` に合わせる。ローカル例: [../../commit-report-tool/patent-digest-handoff.md](../../commit-report-tool/patent-digest-handoff.md)）。

| フィールド | 型 | 必須 | 説明 |
|------------|-----|------|------|
| `applicant` | string | 推奨 | 出願人（企業名など）。 |
| `publication_number` | string | 推奨 | 公開番号／出願番号の表示用。 |
| `ipc` | string | いいえ | IPC 代表コードや要約した分野表現。 |
| `claims_excerpt` | string | いいえ | 請求項の短い抜粋。無ければ省略。 |

特許要約では **権利範囲のニュアンス・出願人・技術的意義**を 2〜4 文のなかで触れる（請求項の全文再現はしない）。

## 3. 出力スキーマ（モデルからの返答）

モデルは **JSON オブジェクトのみ**を返す（前後にマークダウンや説明文を付けない）。

### 3.1 ニュース

```json
{
  "article_id": "string",
  "summary": "string",
  "key_points": ["string", "string", "string"]
}
```

| キー | 制約 |
|------|------|
| `summary` | **日本語**。**新入社員にも通じる説明**を最優先。専門用語は **括弧や短い一文で噛み砕く**。長さは **目安 280〜450 文字**（全角換算）。**最大 2 段落**まで。段落の区切りは **`\n\n`（空行 1 つ）**のみ。見出し記号（`##` 等）は使わない。 |
| `key_points` | **0〜3 要素**。各 **60 文字以内**。補足の短文。根拠が薄い場合は `[]` でよい。 |

**メール／HTML 組み立て**: パイプラインは `summary` 内の `\n\n` を `<p>` 等に分割してよい（モデルに HTML を書かせない方針は変わらない）。

### 3.2 特許

同じキー構造でよい。`summary` は **目安 200〜380 文字**（新入社員向けの解説を含めてよい）、出願人・番号・IPC・権利範囲のいずれかを **誤りなく**1 回以上触れる（入力に無い項目は書かない）。

### 3.3 禁止事項（ハルシネーション・品質）

- 入力にない **数値・固有名詞・因果・法的有効性**を捏造しない。
- **別 URL・別タイトル**を生成しない。リンクはパイプライン側でテンプレに埋め込む想定のため、モデル出力に raw URL を含めてもよいが、**入力の `url` と異なる URL を出してはならない**。
- 憶測は **「〜の可能性がある」**に限定し、1 要約あたり 1 回まで。
- 煽りタイトル、露骨なクリックベイト、**個人・特定企業への中傷・差別**、投資助言は禁止。
- **口調**: **フレンドリー**（「です・ます」でも「だ・である」でも可だが、一要約内で混在させない）。**軽いユーモアや皮肉**は **業界・相場・制度・ニュースの取り上げ方**に対する距離感に限定し、1 要約あたり **軽い一文**程度まで（職場配信として品位を落とさない）。

### 3.4 トーン（ニュース・業界ダイジェスト）

- **読者**: 化学・材料まわりの **新入社員〜中堅**が読んでも追いつける密度。前提知識があっても読みやすいことを損なわない。
- **目的**: 事実関係を正確に伝えつつ、「同僚にメールで共有する」距離感の **親しみやすさ**。
- キャラクター名や特定のキャラ口調の指定はプロンプトに **書かない**（モデル差分を抑える）。

## 4. 関連度 ★3 / ★4 / ★5 と `data-relevance-stars`

### 4.1 意味（既存ダイジェストのレジェンドと整合）

既存メール HTML（例: `output/digest.html`）の説明と揃える。

| `relevance_stars` | 意味（ユーザー向けコピー） | UI 色（静的ページ） |
|-------------------|---------------------------|---------------------|
| 5 | ドンピシャ（関心テーマに直結） | 橙 `orange-*` |
| 4 | 強く関連 | 青 `blue-*` |
| 3 | 部分的関連（配信閾値の下限） | 緑 `green-*` |

配信ポリシー: **★3 未満は配信しない**（パイプラインで除外）。本仕様の HTML 生成対象は **3・4・5 のみ**。

### 4.2 パイプラインから星への写像（数値スコアがある場合）

キーワード／Embedding 等で **0.0〜1.0 の `relevance_score`** があるときの **デフォルト案**（チューニング可。仕様として「単調増加」であることのみ必須）。

| `relevance_score` | `relevance_stars` |
|-------------------|-------------------|
| `>= 0.82` | 5 |
| `>= 0.55` | 4 |
| `>= 0.35`（かつ配信対象に残っている） | 3 |

閾値が無い旧ロジック（整数星のみ）の場合は **その整数をそのまま** `relevance_stars` に用い、3 未満は配信前に除外。

### 4.3 LLM と星の整合

- 星は **パイプラインが決定**する。モデルは `relevance_stars` を **参照**し、★5 なら主題を正面から、★3 なら「周辺だが注目点は…」と **過大評価しない**。
- モデル出力 JSON に星を含めない（単一の真実源をパイプラインに保つ）。

## 5. HTML 契約（テンプレ＋後処理）

### 5.1 各カードのルート

- 要素: `<details>`
- 必須属性: `data-relevance-stars="3"` または `"4"` または `"5"`（文字列）。
- `class`: 星に応じて **枠＋リング**（`patent-digest-handoff-page.html` と同型）。

| 星 | `details` の `class`（抜粋） |
|----|-------------------------------|
| 3 | `rounded-2xl border border-green-200 bg-white shadow-sm ring-1 ring-green-200/80` |
| 4 | `... border-blue-200 ... ring-blue-200/80` |
| 5 | `... border-orange-200 ... ring-orange-200/80` |

### 5.2 `<summary>` 内のピル（Komineko 型）

星ごとに **淡背景＋丸ドット＋★表記**。本文説明は `text-zinc-600`。

- ★3 ピル: `bg-green-100` `text-green-600`、ドット `bg-green-500`、表示 `★★★`
- ★4 ピル: `bg-blue-100` `text-blue-600`、ドット `bg-blue-500`、表示 `★★★★`
- ★5 ピル: `bg-orange-100` `text-orange-600`、ドット `bg-orange-500`、表示 `★★★★★`

具体マークアップは [output/patent-digest-handoff-page.html](../output/patent-digest-handoff-page.html) の `<details data-relevance-stars="...">` ブロックを **コピー基準**とする。

### 5.3 展開部

- コンテナ: `<div class="border-t border-{color}-100 px-4 pb-4 pt-2 sm:px-5 sm:pb-5">`（色は星に合わせる）
- 要約段落: `text-sm leading-relaxed text-zinc-600`
- リンク一覧は既存テンプレ同様 `<ul>` / `<a>` でよい。

**方針**: 星に応じた色クラスは **Python 等のテンプレ**で機械的に出し、LLM には渡さない。

## 6. プロンプト草案（全文）

以下を **システム** と **ユーザー** に分割して利用する。モデルは **JSON のみ**返すこと。

### 6.1 システム（ニュース）

```
あなたは化学・石油化学・材料業界向けニュースダイジェストの編集者である。
入力は JSON で与えられる。各記事について、指定のスキーマで日本語の要約だけを返す。

読者と説明の仕方:
- 化学・材料まわりの新入社員にも通じるように書く。専門用語は初出で短く噛み砕く（括弧や一文）。
- 口調はフレンドリーでよい。業界ニュースや相場・制度などへの軽いユーモアや皮肉は1要約につき軽い1文まで。個人や特定企業を貶す表現は禁止。

厳守:
- 出力は有効な JSON オブジェクト1つのみ。コードフェンスや説明文を付けない。
- 入力にない事実・数値・固有名詞を追加しない。推測は「〜の可能性がある」形式に1回までに限る。
- summary は日本語。目安280〜450文字。段落は最大2つ。段落の区切りは \n\n のみ。key_points は0〜3個。各制限に従う。
- relevance_stars の値に応じたトーン: 5=主題として正面から、4=関連が強い、3=周辺的だが読者の関心に触れる点を過大評価しない。星の数値自体は出力に含めない。
- 入力の url と異なる URL を生成しない。
```

### 6.2 ユーザー（ニュース）— テンプレート

```
次の記事を要約せよ。返答スキーマ:
{"article_id":"<同じ>","summary":"...","key_points":[]}

記事JSON:
{{ARTICLE_JSON}}
```

実装では `{{ARTICLE_JSON}}` を 1 件分のオブジェクトに置換する。バッチ処理なら記事ごとに **別リクエスト**推奨（品質と失敗時の再試行のため）。

### 6.3 システム（特許）— ニュースとの差分

ニュース用システムの末尾に **次の段落を追加**する。

```
特許入力の場合: summary は出願人・公開／出願番号・IPC のうち入力にあるものを誤りなく触れ、
権利範囲や技術的意義を説明する。請求項の全文は書かない。目安200〜380文字。段落は最大2つ（\n\n）。
新入社員向けの用語補足とフレンドリーなトーンはニュースと同様。軽い皮肉は対象を特許制度・技術トレンドに限定する。
```

### 6.4 ユーザー（特許）

ニュースと同じ `{{ARTICLE_JSON}}` テンプレートでよい。フィールドが増えたオブジェクトを渡す。

## 8. 仕様確定後の実装フェーズ（メモのみ）

- `google-genai` 等で上記 JSON I/O を実装し、`chem-digest-tool` の `filter.py` パターンを流用する。
- 収集・スコアリング復元後、`output/digest-YYYY-MM-DD.html` を **本テンプレの `<details>`** で組み立てる。
- GitHub Actions / `requirements.txt` / `src/*.py` の復元と CI。
- 手動参照 `output/patent-digest-handoff-page.html` を生成ロジックに取り込むかは実装時に選択。
